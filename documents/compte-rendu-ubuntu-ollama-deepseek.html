Compte-rendu — Mise en place d’un serveur Ubuntu (OpenSSH) et d’Ollama / DeepSeek

BTS SIO — SISR & usage projet SLAM (API locale) — travaux pratiques

Contexte

Installation d’Ubuntu Server (version sans interface graphique), puis du service OpenSSH pour administration à distance. Paramètres réseau cibles :

  • Adresse IP : 10.11.11.10/16
  • Passerelle : 10.11.0.1
  • DNS : 10.10.0.1
  • Compte principal : admin1 (mot de passe : Admin1)

Cette machine sert de plateforme pour héberger localement une version de DeepSeek via Ollama. Objectif : permettre aux SISR d’envisager du RAG sur le matériel des salles (S2, S4) et aux SLAM d’appeler une IA par API sans abonnement externe.

Vérification matérielle / stockage

Contrôle du RAID 5 : vérification que le stockage cible le mode souhaité (interface matérielle, puis pendant l’installation Subiquity / Curtin notamment via la prise en charge mdadm — cf. figure 2 : ligne configuring raid (mdadm) service dans le journal d’installation).

Étape 1 — Installation Ubuntu Server et service OpenSSH

Installation depuis une clé USB bootable (serveur HPE ProLiant) : choix de la base Ubuntu Server (pas la variante minimisée afin d’avoir un serveur prêt à l’emploi), partitionnement (LVM, etc. selon l’installateur), puis mises à jour de paquets et enfin vérification / démarrage d’OpenSSH.

Résumé — Étape 1

Ubuntu Server a été installé sur le matériel physique via clé USB bootable, la pile réseau a été préparée (IP statique : voir consignes), OpenSSH a été installé et le compte admin1 créé (mot de passe : Admin1). Le service SSH a été démarré, activé au boot et vérifié.

Étape 2 — Paire de clés SSH, dossier ~/.ssh et authorized_keys

Premières tentatives (souvent en root) : ssh-keygen (clé ed25519 par défaut) avec chemin cible = répertoire seul — erreur Is a directory ; cat id_rsa.pub >> authorized_keys — fichier manquant. En compte admin1, l’adresse 10.11.11.10 apparaît sur l’interface eno2np1 (fig. 9). Ensuite : création de /home/admin1/.ssh en chmod 700, génération de la paire avec un chemin de fichier explicite, alimentation de authorized_keys et ls -l (figures 10 et 11).

Erreur usuelle tant que la paire n’existe pas : cat: /home/admin1/.ssh/id_rsa.pub: No such file or directory.

Résumé — Étape 2

Paire de clés SSH générée pour admin1, authorized_keys alimenté, permissions sécurisées. Le service SSH est opérationnel côté serveur ; le test client complet sera effectué lorsque le poste client sera sur le même réseau.

Étape 3 — Ollama et modèle DeepSeek (CPU)

Ollama installé, unité systemd active, API exposée en local sur 127.0.0.1:11434. Fonctionnement en CPU uniquement, cohérent avec le matériel. Téléchargement volumineux du modèle DeepSeek, lancement en ligne de commande, tests de dialogue ; sortie avec Ctrl + D.

Étape 4 — Ouvrir l’API Ollama sur l’adresse du serveur (override systemd)

Par défaut, Ollama n’écoute que 127.0.0.1. Mise en place d’un override systemd, par ex. :

sudo mkdir -p /etc/systemd/system/ollama.service.d
sudo nano /etc/systemd/system/ollama.service.d/override.conf

Fichier d’override typique (variable selon la version) pour lier l’écoute à l’interface du serveur, par exemple (contenu reprenant la capture) : 

[Service]
Environment="OLLAMA_HOST=0.0.0.0:11434"

Étape 5 — Pare-feu, SSH en local puis tests réseau

Autorisation de SSH dans le pare-feu (ex. ufw), rechargement / redémarrage du service, test ssh admin1@localhost, puis depuis un client : ping et connexion SSH.

Étape 6 — Clé SSH côté client et intégration sur le serveur

Génération de la clé sur le poste client, affichage de la clé publique pour copie sur le serveur. Correction d’une erreur (mauvaise clé collée), puis ajout de la bonne clé sur la deuxième ligne de ~/.ssh/authorized_keys, correctifs des droits (fichier et répertoires), ajustement éventuel de /etc/ssh/sshd_config, redémarrage de SSH et dernière vérification des permissions.

Étape 7 — DeepSeek / Ollama accessibles sur le réseau (client)

Vérification de l’écoute de l’API ; ajout d’une ligne dans la section [Service] du fichier /etc/systemd/system/ollama.service si la configuration a été complétée en complément (selon la capture) ; rechargement et redémarrage des services.

Test HTTP depuis le client (réponse JSON attendue). Le pare-feu bloquait initialement : ouverture explicite avec sudo ufw allow 11434/tcp puis sudo ufw reload.

Résumé — Étape 7

Le serveur héberge DeepSeek via Ollama. L’API HTTP est exposée sur le réseau (IP du serveur) et atteignable depuis les postes clientes, ce qui ouvre l’intégration (SLAM, scripts) sans abonnement externe.

Étape 8 — Consolidation, reboot et correction Netplan

Vérification du redémarrage propre (sudo reboot) et de l’état des services (systemctl status sur ollama et ssh selon besoin).

Problème constaté : plus d’IPv4 après reboot, interfaces down (commande ip a, ping KO). Dépôt netplan : ls /etc/netplan, édition du fichier YAML concerné sudo nano /etc/netplan/*.yaml, application avec sudo netplan generate et sudo netplan apply (ou équivalent), puis vérification ip a et ip route pour 10.11.11.10/16 et la passerelle.

Exemple de structure attendue (à adapter exactement sur la forme de votre interface, ex. ens18) :

network:
  version: 2
  ethernets:
    ensXX:
      addresses: [10.11.11.10/16]
      routes:
        - to: default
          via: 10.11.0.1
      nameservers:
        addresses: [10.10.0.1]

Étape 9 — Interface web (Apache)

Insertion d’une clé USB contenant les fichiers de l’interface, montage du périphérique, installation d’Apache2, copie (ou déplacement) des fichiers de l’interface dans /var/www/html afin d’en faire le document root du site par défaut.

Conclusion

Le serveur Ubuntu fournit SSH sécurisé (clés, pare-feu), héberge Ollama / DeepSeek en local avec API joignable sur le LAN, et peut servir une interface web légère via Apache. Cette base permet aux travaux SISR / SLAM d’itérer sur le RAG et les appels API sans dépendre d’un fournisseur payant, sous réserve de la charge CPU et d’une politique de sauvegarde / supervision adaptées.
